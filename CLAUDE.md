# 공부사주 (가칭) — 작업 규칙

공부·진로 사주 리포트 유료 서비스. 웹 우선, "신청 → (검수) → 전달" 모델.
사주 해석(믿음 기반)과 학교 데이터(공공데이터 사실)를 **분리**한 채 하나의 리포트로 합친다.
상세 설계는 `@SPEC.md` 참고.

## 절대 규칙 (load-bearing)

1. **2레이어 분리(코드 강제)**: `lib/saju` ↔ `lib/schools` 상호 import 금지. 둘을 합치는 곳은 오직 `lib/report`. ESLint import 경계 룰로 차단한다.
2. **LLM은 학교 사실을 만들거나 바꾸지 않는다**: 학교명·진학률 등 사실은 코드가 템플릿에 삽입. LLM은 해석·연결 산문만 작성.
3. **금지 표현**: 단정·보장("이 학교 가면 된다"), 사주→특정 학교를 "정답"으로 연결하는 인과 표현 금지 (표시광고법 리스크). "예측/최적화" 대신 "참고/경향/해석".
4. **학교 배정은 항상 "예상 배정(교육청 확인 필요)"**로 표기하고 출처·기준일을 붙인다.
5. **만세력은 동경 127.5° 기준 경도 보정(-30분) 적용.** KST(동경 135°) 입력 → `applyTrueSolarTime`으로 -30분 보정 후 時柱 결정. 기본 ON, 토글 없음. 균시차·출생지 경도는 미적용(한국 주류 만세력 관행).
6. **만세력 엔진은 테스트 먼저**: 알려진 만세력 결과를 픽스처로 박고 통과 후 진행. 절기 기준 월주 / 진태양시 보정 / 서머타임 / 시간 모름 케이스 반드시 처리.
7. **데이터 줄세우기 금지**: 학교별 학업성취 등급·순위, 상위권 대학 진학률은 공개되지 않으며 가공·단정하지 않는다.
8. **PII**: 생년월일시·주소·학교는 암호화·분리 저장, 법정대리인 동의, 보관기간 명시.
9. **data-pipeline은 앱과 분리**: 학교·통학구역 데이터는 별도 배치로 적재(3·9월 갱신). 런타임은 조회만.

## 스택

Next.js 15 (App Router, TS) · Supabase Postgres + PostGIS · Prisma · Claude API(Sonnet) · 토스/카카오페이 · 카카오 알림톡·이메일

## 작업 방식

- 모듈 단위로 "구현 → 테스트 → 다음". 한 번에 전체 X.
- 빌드 순서는 `@SPEC.md`의 "8. 빌드 순서" 를 따른다.
- 가장 위험한 만세력 엔진(`lib/saju`)부터 테스트와 함께 검증한다.
