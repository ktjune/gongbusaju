# SPEC — 공부사주 (가칭) 서비스 설계

> 공부·진로 사주 리포트 유료 서비스. 웹 우선, "신청 → (검수) → 전달" 모델.
> 사주 해석(믿음 기반)과 학교 데이터(공공데이터 사실)를 **분리**한 채 하나의 리포트로 합친다.
> 매 세션 적용할 핵심 규칙은 `CLAUDE.md` 참고. 이 문서는 상세 설계.

---

## 1. 제품 개요

- **무엇**: 자녀 생년월일시 기반 공부·기질 사주 리포트. 상위 요금제는 거주지/현재 학교 기반 학교군 정보 큐레이션까지.
- **채널**: 별도 앱 X. 웹(랜딩 → 신청·결제 → 결과 페이지 + PDF). 알림은 카톡/이메일.
- **포지셔닝**: "공부사주/입시사주" 키워드. 가격 앵커는 점집 5만원대.

### 요금제

| | Basic | Premium |
|---|---|---|
| 내용 | 공부·기질 사주 리포트 (해석) | Basic + 지역 학교군 정보 큐레이션 + 기질 관점 병기 |
| 입력 | 생년월일시·성별 | + 현재 학교 / 주소 |
| 데이터 | 만세력 계산 | + 통학구역·진학현황·학교 기본정보 |
| 성격 | 미끼·신뢰 구축 | 차별화·실수익 |

## 2. 핵심 설계 원칙 — 2레이어 분리

제품 신뢰도와 법적 안전(표시광고법)의 핵심. 해석과 사실을 **코드/DB/UI에서 물리적으로 분리**한다.

| | A. 사실 레이어 | B. 해석 레이어 |
|---|---|---|
| 내용 | 배정 예상 학교·통학거리·학교 유형·진학현황 | 사주 오행·기질·대운 |
| 출처 | 공공데이터 (출처·기준일 표기) | 만세력 계산 + LLM 해석 |
| 표기 | 사실 / "예상 배정, 교육청 확인 필요" | "해석·참고용" 명시 |

- 두 레이어는 서로의 출력을 입력으로 받지 않는다.
- 결합("오행이 X라 이 학교가 정답")은 시스템이 **하지 않는다.** 두 블록을 나란히 제시하고 판단은 부모 몫.

## 3. 시스템 파이프라인 (비동기)

```
[주문] 생년월일시·성별·(Premium)주소/학교·tier
   │
   ├── 해석 레이어 ──────────────┐     ┌── 사실 레이어 (Premium) ────────┐
   ▼                             │     ▼                                │
[1 만세력 엔진] 결정론적         │   [3 학교데이터 엔진]                │
  양력↔음력+절기 → 사주팔자      │     주소→지오코딩→좌표               │
  오행%·십성·대운 (JSON)         │     →통학구역 point-in-polygon       │
   │                            │      →학교군 사실테이블(출처·기준일)  │
   ▼                            │      │   ※ 사주를 입력으로 받지 않음   │
[2 기질 매핑] 규칙표            │      │                                │
  오행/십성→레이더 점수(JSON)    │     │                                 │
   └─────────────┬──────────────┘      └────────────┬───────────────────┘
                 ▼                                   ▼
           [4 리포트 생성기 (LLM)] ← 학교 사실은 코드가 삽입, LLM은 못 바꿈
             구조화 출력 → 템플릿(사실 블록 / 관점 블록 분리)
                 ▼
           [5 검수] 초기 사람 → 이후 금지표현 classifier 자동화
                 ▼
           [6 발행] 웹 결과페이지(토큰)+PDF → 카톡/메일 알림
```

"신청→제작→전달" 모델이므로 제작은 **비동기 잡(큐)**으로 처리한다.

## 4. 레포 / 모듈 구조

폴더 구조 자체가 2레이어 분리를 강제한다.

```
/app
  /(public)        랜딩 · /apply 신청·결제 · /result/[token] 결과
  /admin           검수 큐
  /api             order · payment-webhook · generate-trigger
/lib
  /saju            ← 해석 레이어 (결정론, LLM 없음)
    calendar.ts      양력↔음력 · 절기
    pillars.ts       사주팔자 4기둥
    elements.ts      오행 % · 십성
    daeun.ts         대운
    traits.ts        기질 매핑 규칙표
    index.ts         → SajuResult
  /schools         ← 사실 레이어 (saju import 금지)
    geocode.ts       주소 → 좌표
    zone.ts          PostGIS point-in-polygon
    query.ts         학교군 조인
    index.ts         → SchoolFacts
  /report          ← 유일하게 두 레이어를 import 하는 곳
    template.ts      사실 블록 / 관점 블록 템플릿
    generate.ts      Claude 호출 (좁은 역할)
    guardrails.ts    금지표현 검사
  /pdf  /notify
/db                prisma schema + migrations
/workers           비동기 제작 잡 (큐 소비)
/data-pipeline     학교·통학구역 적재 배치 (앱과 분리, 3·9월 갱신)
```

**강제 규칙**: `lib/schools` ↔ `lib/saju` 상호 import 금지. 합치는 곳은 오직 `lib/report`. ESLint import 경계 룰로 차단.

## 5. 모듈 계약 (경계만 — 구현은 Claude Code에서)

```ts
// /lib/saju — (생년월일시+성별) 해시로 캐시, PII 분리
type SajuResult = {
  pillars: { year; month; day; hour };       // 4기둥 간지
  elements: { 목: number; 화: number; 토: number; 금: number; 수: number }; // %
  tenGods: Record<string, number>;            // 십성 분포
  daeun: { age: number; ganji: string }[];     // 대운
  traitScores: Record<string, number>;         // 레이더 6축 (해석 지표, 측정 아님)
};

// /lib/schools — 사주 안 받음. 주소만.
type School = { name; type; distanceM; admissionStats?; source; asOf };
type SchoolFacts = {
  assignedSchool?: School;   // "예상 배정(교육청 확인 필요)"
  cluster: School[];
  source: string; asOf: string;
};

// /lib/report — 합치는 입력
type ReportInput = { saju: SajuResult; schools?: SchoolFacts; tier: 'basic' | 'premium' };
// 학교 사실은 코드가 템플릿에 박고, LLM엔 "관점 산문"만 시킨다.
```

## 6. 데이터 모델 (Supabase + PostGIS / Prisma)

| 테이블 | 역할 | 주의 |
|---|---|---|
| `orders` | 주문·결제·상태 | status: paid→generating→review→published |
| `subjects` | 자녀 PII(생년월일시·주소·학교) | **암호화·분리 저장**, 보관기간·동의 |
| `saju_results` | 사주 계산 캐시 | 생년월일시 해시 키 (PII 분리) |

| `schools` | 학교 사실·진학현황 | 출처·기준일 컬럼 필수 |
| `reports` | 생성물·토큰·PDF·검수상태 | token 으로 결과페이지 접근 |

PII(`subjects`)와 계산결과(`saju_results`, `reports`)는 분리한다.

### 해석 레이어 (외부 수급 아님)

| 용도 | 방식 |
|---|---|
| 사주팔자·오행·대운 | 오픈소스 만세력 라이브러리 자체 계산 (예: lunar-javascript) |
| 리포트 문장·점수 | Claude API |

### 라이선스 핵심

- 공공데이터포털 표준데이터는 **데이터셋별로 공공누리 유형 직접 확인** (상업 이용 가부).
- 학구도 GIS는 "참조용·법적효력 없음" → **"예상 배정(교육청 확인 필요)"**로만 표기.
- 일부 지자체 자체 통학구역(예: 서울 스마트서울맵)은 **상업 이용 금지** → 표준데이터를 우선 사용.
- 학교·학구도 표준데이터는 매년 **3월·9월** 갱신 → 배치 주기 반영.

### 데이터로 가능/불가 (중요)

- ⭕ 주소→배정 학교, 학교 기본정보·교육여건, 중학교 졸업생 진로현황(특목고/자율고 진학), 고교 대학진학률(전반)
- ❌ 학교별 학업성취도 등급·순위, 상위권 대학(SKY 등) 진학률 → 공개 안 됨. **줄세우기/등급화 가공도 하지 않는다.**

## 8. 빌드 순서

1. **스캐폴드**: Next.js + Supabase + Prisma, PostGIS 확장 on.
2. **`/lib/saju` + 테스트 먼저**: 가장 위험. 알려진 만세력 결과를 테스트 픽스처로 박고 통과 후 진행.
3. **`/data-pipeline` + `/lib/schools`**: 통학구역·학교 적재 → point-in-polygon 조회.
4. **`/lib/report`**: 템플릿 + LLM(좁은 역할) + guardrails.
5. **주문/결제/결과페이지/admin 검수**.
6. **알림(카톡·메일) + PDF**.

모듈 단위로 "구현 → 테스트 → 다음". 한 번에 전체 X.

---

## 9. 어려운 3곳 (여기에 시간 투자)

1. **만세력 정확도**
   - 사주 월주는 음력이 아니라 **24절기(立春 기준)**로 정함.
   - **진태양시 보정**: 한국 표준시 동경 135° vs 실제 ~127° → 약 30분 차이로 時柱 변동.
   - **서머타임 보정**: 과거 일부 연도 출생자 시각 보정 (적용 연도 확정 후 구현).
   - **시간 모름**: 時柱 제외 모드 (지어내지 않기).
2. **통학구역 point-in-polygon**: 통학구역이 폴리곤(도면)이라 PostGIS 필수. 텍스트 매칭 불가.
3. **LLM이 학교 사실을 못 건드리게**: 학교명·숫자는 코드가 템플릿에 삽입, LLM은 연결 산문만. 환각 차단 = 법적 분리 강제.

## 10. 법적 / PII 제약

- **표시광고법**: 단정·보장 표현 금지. 사주→특정 학교 인과 결합 금지. "예측/최적화" 대신 "참고/경향/해석".
- **개인정보보호법**: 미성년 자녀 생년월일시·주소·학교 수집 → 법정대리인 동의, 수집 최소화·암호화, 보관기간 명시.
- **전자상거래법**: 디지털 콘텐츠 환불 정책 — "제작 착수 전 취소 가능" 구조 권장.
- **카카오 알림톡**: 정보성·템플릿 심사 대상. 카톡은 "완성 + 링크" 알림 채널, 본문은 웹/PDF.

## 11. 전달 모델

```
신청 폼(생년월일시·주소·학교·연락처) → 결제 → 백엔드 생성 → (초기)사람 검수
   → 웹 결과 페이지(토큰 링크) + PDF → 카톡/메일 완성 알림
```

- 즉시 출력 X. 검수 버퍼로 품질·법적 표현·학교 사실을 거른다. 안정화되면 classifier로 자동화.
- 일회성 구매지만 "한 아이 인생에서 여러 번"(형제 추가, 초→중→고 전환기, Basic→Premium) → **연락처 확보가 핵심 자산.**
